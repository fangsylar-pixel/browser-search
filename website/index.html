Why agents use it

Browser Search MCP is designed for agent workflows where search snippets are not enough and the caller needs an inspectable local tool.

Real browser search

Search through Chrome/Edge instead of relying only on HTTP scraping or a hosted search API.

Page reading

Open top results and return cleaned page text, title, final URL, description, and publish metadata.

Query planning

Use web_search_plan to expand vague user questions into better candidate queries.

Quality diagnostics

web_research reports strict or partial coverage and tells callers which anchors were missing.

No API key by default

Browser mode works locally without a search API key, with optional Tavily and Brave providers.

Authenticated workflows

Optional browser-takeover bridge support can reuse an existing browser session.

Plan before searching

Agents can inspect intent and candidate queries before starting a browser search.

{
  "query": "home projector vs TV which is better",
  "intent": {
    "topic": ["projector", "TV"],
    "task": ["compare", "buying guide", "recommendation"]
  },
  "candidate_queries": [
    "home projector vs TV which is better",
    "projector TV comparison pros cons buying guide",
    "projector TV comparison recommendation"
  ]
}

Return research-ready content

Search, filter, read pages, and return diagnostics in one MCP call.

MCP tools

Use web_research as the default tool for grounded answers.

ToolWhat it does
web_search_planAnalyze intent and candidate queries without launching a browser.
web_searchSearch one engine and return structured results.
web_search_multiSearch multiple engines and optionally deduplicate results.
web_search_read_pageRead structured content from a URL.
web_researchSearch, filter, read pages, and return quality diagnostics.
web_search_statusCheck browser, bridge, provider, and cache status.
